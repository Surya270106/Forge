# ADR 0001: Pre-Release Migration Baseline Reset

**Status**: Accepted  
**Date**: 2026-07-20  

## Context
During the environment recovery and integration verification phase, it was identified that the original manual Alembic migration definitions (`007_enterprise_auth.py`, `008_tenant_rls.py`, etc.) were diverging significantly from the SQLAlchemy ORM models, causing migration failures (e.g., missing `organization_id` columns and conflicting ENUM declarations). 

Because the system was heavily broken in local development environments and could not even start up against a clean PostgreSQL instance, a decision was made to reset the migration history.

## Decision
We elected to perform a **pre-release migration baseline reset**. 
1. We deleted all existing `migrations/versions/*.py` files.
2. We dropped the entire `forge` PostgreSQL database.
3. We ran `alembic revision --autogenerate` to establish a single, comprehensive `56af2791774f_initial_schema.py` migration containing the complete current schema (tables, constraints, foreign keys, indexes).
4. We manually created a follow-up migration `d7b5578b4249_add_tenant_rls_policies.py` to explicitly enforce Row-Level Security (RLS) on all correct tenant tables.

## Justification & Safety
This destructive action is deemed safe and acceptable **only** because:
- **No Shared Deployment**: This repository has never had an external, persistent, or shared staging/production deployment.
- **No Legacy Data**: No user or production database depends on the previous revision IDs.
- **Pre-Alpha Stage**: The reset occurred before the Internal Alpha acceptance milestone.

## Verification
- We verified the resulting autogenerated migration comprehensively. PostgreSQL types, indexes, foreign keys, and constraints perfectly match the source of truth in `packages/database/models/`.
- We ensured RLS policies are located in an explicit, manually reviewed migration rather than relying solely on autogeneration.

## Consequences & Future Rules
1. **Immutability**: The pre-release migration baseline is now frozen. From this point forward:
   - never regenerate the full migration history
   - never edit an already shared migration revision
   - every schema change requires a new migration
   - every RLS change requires an explicit reviewed migration
   - CI must verify migration from a clean PostgreSQL database
